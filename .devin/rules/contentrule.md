# Portfolio Content Rules

## Primary Objective

Improve the content of the portfolio website while preserving authenticity.

The website should represent a real software engineer, not a marketing brochure.

Every change must make the content:
- clearer
- more readable
- slightly more professional
- more engaging

without sounding AI-generated.

---

# Writing Style

Write naturally.

Avoid:
- corporate buzzwords
- excessive adjectives
- exaggerated claims
- unnecessary jargon
- dramatic language

Prefer:

Instead of:
"Passionate, highly motivated software engineer delivering cutting-edge AI-driven scalable enterprise solutions."

Write:
"Software Engineer building AI-powered support applications and internal tools."

The writing should feel like a person wrote it.

---

# Tone

Use a tone that is:

- confident
- professional
- approachable
- concise
- genuine

Never sound arrogant.

Never oversell experience.

Never invent achievements.

---

# Keep It Human

The portfolio should be easy to skim.

Visitors should understand each section in a few seconds.

Avoid long paragraphs unless absolutely necessary.

Use:

- short paragraphs
- concise descriptions
- meaningful headings
- natural transitions

---

# Do Not Inflate Experience

Do not change:

- job titles
- responsibilities
- technologies
- achievements
- dates
- projects

Only improve wording.

Never exaggerate impact.

Never invent metrics.

Never add numbers unless already provided.

---

# Projects

Every project should answer:

1. What is it?
2. Why was it built?
3. What technologies were used?
4. What was your contribution?

Avoid lengthy explanations.

Focus on clarity.

---

# Experience

Convert bullet points into concise professional descriptions where appropriate.

If bullets are easier to read, keep them.

Do not force paragraphs.

---

# Skills

Group related skills together.

Prefer categories such as:

- Languages
- Frontend
- Backend
- AI
- Databases
- Cloud
- DevOps
- Tools

Avoid huge comma-separated lists.

---

# About Me

The About section should feel personal without being overly casual.

Good topics:

- what you currently work on
- what interests you
- what you enjoy building
- what you are learning

Avoid clichés like:

- passionate developer
- coding enthusiast
- technology lover
- lifelong learner

---

# Hobbies

Keep hobbies simple.

Do not elaborate unnecessarily.

Example:

Football

instead of

"I have a deep passion for football and enjoy competing whenever possible."

---

# Activities

Activities should explain participation naturally.

Example:

"Part of Dell's Football Club"

instead of

"Actively engaged as a valuable member of Dell's Football Club."

---

# Resume Consistency

Portfolio content should stay consistent with the resume.

If information differs,
ask before changing it.

---

# Grammar

Fix:

- grammar
- punctuation
- repetition
- awkward wording
- inconsistent tense

Do not rewrite content that is already good.

---

# Brevity

Shorter is usually better.

If two sentences can become one, do it.

If one sentence communicates enough, stop there.

---

# SEO

Improve headings where appropriate.

Do not stuff keywords.

Natural language is preferred.

---

# Accessibility

Prefer:

clear headings

meaningful section titles

short paragraphs

descriptive button labels

simple language

---

# Review Rule

Before making changes, ask:

Does this sound like a real engineer?

Would I naturally say this in an interview?

If not,
rewrite it.

---

# Final Rule

The goal is refinement, not reinvention.

Improve the writing while keeping the author's personality intact.
